1. Resumen ejecutivo
✅
Qué se construye
Tres capas nuevas sobre el producto: (a) categorías jerárquicas globales para clasificar tipo de producto, (b) atributos globales con valores predefinidos como sistema (tallas, colores), y (c) variantes por producto = combinación de valores que tiene stock propio e imagen propia.
⚡
Por qué global y no por negocio
Adifnex es SaaS. Si cada negocio crea sus propias tallas, habrá 50 formas distintas de escribir "XL". Estandarizar en el sistema permite filtros cruzados, reportes y consistencia. El negocio solo elige qué valores aplica a su producto — no los crea.

5. Colores + imágenes por color
🎨
Cómo funciona
  1. Al configurar el atributo Color en un producto, el negocio elige los colores disponibles.
  2. Para cada color elegido, puede subir una imagen.
  3. En el catálogo público, el cliente selecciona el color → la imagen principal cambia automáticamente.
  4. Si un color no tiene imagen propia, se muestra la imagen principal del producto.

6. Modelo de datos completo (SQL)
💡
Enfoque de migración
8 tablas nuevas. Ninguna tabla existente se modifica estructuralmente (producto, stock_sede, movimiento_inventario se mantienen). Solo se agregan relaciones opcionales. Los productos sin variantes siguen funcionando igual.
Tabla: categoria — Árbol jerárquico global
CREATE TABLE categoria (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    nombre                VARCHAR(100) NOT NULL,
    padre_id              BIGINT REFERENCES categoria(id),  -- null = raíz
    slug                  VARCHAR(100) UNIQUE NOT NULL,   -- 'ropa-mujer-exterior'
    icono                 VARCHAR(50),                     -- 'pi-shopping-bag'
    orden                 INTEGER NOT NULL DEFAULT 0,
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    estado                BOOLEAN NOT NULL DEFAULT true,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE INDEX idx_categoria_padre ON categoria(padre_id) WHERE eliminado = false;
Tabla: atributo_tipo — Tipos de atributo (global)
CREATE TABLE atributo_tipo (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    nombre                VARCHAR(100) NOT NULL,        -- 'Talla ropa adulto', 'Color'
    slug                  VARCHAR(100) UNIQUE NOT NULL,   -- 'talla-ropa-adulto', 'color'
    tipo_dato             VARCHAR(20) NOT NULL,          -- 'lista' | 'color' | 'texto' | 'numero'
    afecta_imagen         BOOLEAN NOT NULL DEFAULT false,  -- true solo en Color
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    estado                BOOLEAN NOT NULL DEFAULT true,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
Tabla: atributo_valor — Valores de atributo (seed global)
CREATE TABLE atributo_valor (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    atributo_tipo_id      BIGINT NOT NULL REFERENCES atributo_tipo(id),
    valor                 VARCHAR(50) NOT NULL,    -- 'XS', '35', '#EF4444'
    etiqueta              VARCHAR(100),            -- 'Extra Small', 'Talla 35', 'Rojo'
    orden                 INTEGER NOT NULL DEFAULT 0,
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    estado                BOOLEAN NOT NULL DEFAULT true,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE UNIQUE INDEX uq_atributo_valor ON atributo_valor(atributo_tipo_id, valor);
Tabla: categoria_atributo_tipo — Qué atributos sugiere cada categoría
CREATE TABLE categoria_atributo_tipo (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    categoria_id          BIGINT NOT NULL REFERENCES categoria(id),
    atributo_tipo_id      BIGINT NOT NULL REFERENCES atributo_tipo(id),
    requerido             BOOLEAN NOT NULL DEFAULT false,
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE UNIQUE INDEX uq_cat_atrib ON categoria_atributo_tipo(categoria_id, atributo_tipo_id);
Tabla: producto_categoria — Categoría de cada producto
CREATE TABLE producto_categoria (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    producto_id           BIGINT NOT NULL REFERENCES producto(id),
    categoria_id          BIGINT NOT NULL REFERENCES categoria(id),
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE UNIQUE INDEX uq_prod_cat ON producto_categoria(producto_id) WHERE eliminado = false;
-- Un producto pertenece a una sola categoría (hoja del árbol)
Tabla: producto_variante — Combinación de valores para un producto
CREATE TABLE producto_variante (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    producto_id           BIGINT NOT NULL REFERENCES producto(id),
    negocio_id            BIGINT NOT NULL,           -- tenant isolation
    sku_variante          VARCHAR(150),              -- 'POLO-M-BL' (nullable, auto-generado)
    precio_override       NUMERIC(10,2),             -- null = usa precio del producto base
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    estado                BOOLEAN NOT NULL DEFAULT true,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE INDEX idx_variante_producto ON producto_variante(producto_id, negocio_id) WHERE eliminado = false;
Tabla: producto_variante_valor — Qué valores tiene cada variante
CREATE TABLE producto_variante_valor (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    variante_id           BIGINT NOT NULL REFERENCES producto_variante(id),
    atributo_valor_id     BIGINT NOT NULL REFERENCES atributo_valor(id),  -- ej: valor 'M' o 'Rojo'
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE UNIQUE INDEX uq_variante_valor ON producto_variante_valor(variante_id, atributo_valor_id);
Tabla: producto_color_imagen — Imagen por color de producto
CREATE TABLE producto_color_imagen (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    producto_id           BIGINT NOT NULL REFERENCES producto(id),
    negocio_id            BIGINT NOT NULL,
    atributo_valor_id     BIGINT NOT NULL REFERENCES atributo_valor(id),  -- debe ser de tipo color
    url_imagen            VARCHAR(500) NOT NULL,
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE UNIQUE INDEX uq_prod_color_img ON producto_color_imagen(producto_id, atributo_valor_id) WHERE eliminado = false;
Tabla: stock_variante_sede — Stock por variante y sede
CREATE TABLE stock_variante_sede (
    id                    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    negocio_id            BIGINT NOT NULL,
    variante_id           BIGINT NOT NULL REFERENCES producto_variante(id),
    sede_id               BIGINT NOT NULL REFERENCES sede(id),
    cantidad              INTEGER NOT NULL DEFAULT 0 CHECK (cantidad >= 0),
    usuario_creacion      BIGINT NOT NULL,
    usuario_actualizacion BIGINT NOT NULL,
    fecha_creacion        TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    fecha_actualizacion   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    eliminado             BOOLEAN NOT NULL DEFAULT false
);
CREATE UNIQUE INDEX uq_stock_variante_sede ON stock_variante_sede(variante_id, sede_id) WHERE eliminado = false;
Diagrama de relaciones
[categoria]  (árbol, global, seed)
    └── tiene sugeridos → [categoria_atributo_tipo] → [atributo_tipo]
                                                               └── tiene → [atributo_valor]

[producto] (por negocio, ya existe)
    ├── pertenece a → [producto_categoria] → categoria
    ├── tiene variantes → [producto_variante]
    │                         └── combina valores → [producto_variante_valor] → atributo_valor
    ├── tiene imágenes por color → [producto_color_imagen] → atributo_valor (color)
    └── tiene imágenes generales → [producto_imagen] (ya existe)

[producto_variante]
    └── tiene stock por sede → [stock_variante_sede] → sede
7. Decisiones de diseño clave
🏗️ Decisión 1: ¿Un producto puede ser hijo de múltiples categorías?
A
Multi-categoría (N:M)
Un polo puede estar en "Ropa Hombre" y "Ropa Unisex". Más flexible, queries complejas.
B
Una sola categoría (hoja) RECOMENDADO
Más simple. Si quieren multi-categoría, las etiquetas actuales ya lo resuelven. Categoría = clasificación principal para sugerir atributos.
📐 Decisión 2: ¿Cómo se genera el SKU de una variante?
A
Auto-generado por el sistema RECOMENDADO
El sistema concatena el SKU base del producto + las iniciales de los valores: POLO001-M-BL. El negocio puede editarlo manualmente si quiere.
B
Manual (el negocio ingresa cada SKU)
Máximo control, pero carga de trabajo alta si hay 20 variantes.
💰 Decisión 3: ¿El precio varía por variante?
A
Precio base en producto + override opcional por variante RECOMENDADO
La columna precio_override en producto_variante es nullable. Si es null, hereda el precio del producto. Ropa talla XL puede costar S/2 más. Simple de implementar y escala bien.
B
Precio obligatorio en cada variante
Más preciso pero obliga al negocio a ingresar precio en cada una de las N variantes.
🌐 Decisión 4: ¿Los negocios pueden agregar sus propias categorías?
A
Solo el sistema administra categorías (solo SUPERADMIN)
Máxima estandarización. Requiere más curaduría del árbol inicial.
B
Sistema + negocio puede agregar subcategorías propias RECOMENDADO
Las raíces son del sistema. Un negocio puede crear "Ropa de trabajo" bajo "Ropa Hombre" si lo necesita. Columna negocio_id nullable en categoria — null = global.
📦 Decisión 5: ¿Qué pasa con productos existentes (sin variantes)?
A
Coexistencia — variantes son opcionales RECOMENDADO
Los productos actuales siguen usando stock_sede. Si el negocio quiere variantes, las configura y el sistema usa stock_variante_sede. Migración no destructiva, sin romper nada.
B
Migración forzada — todos los productos generan variante base
Unifor el modelo pero requiere migrar datos existentes y adaptar todos los endpoints. Mayor riesgo.

9. Flujo UX — Cómo lo usa el negocio
➕
Crear un producto con variantes (Nuevo flujo)
1
Datos básicos del producto — igual que ahora (nombre, SKU base, descripción, marca, precio base)
2
Elegir categoría — dropdown con árbol. Al elegir "Ropa Mujer — Exterior", el sistema muestra Atributos sugeridos: Talla ropa adulto, Color.
3
Configurar atributos — el negocio selecciona cuáles valores aplican: Talla: [S, M, L, XL]. Color: [Blanco, Negro, Rojo]. El sistema genera la matriz de variantes automáticamente (S×Blanco, S×Negro… = 12 variantes).
4
Subir imágenes por color — para cada color elegido, puede subir una imagen. Si no sube, hereda la imagen principal del producto.
5
Stock por variante — tabla editable mostrando cada variante generada. El negocio puede ingresar el stock inicial de cada combinación (por sede si aplica).
6
Precio por variante (opcional) — si alguna talla o color tiene precio diferente, puede sobreescribirlo. Si no, hereda el precio base.
🔍
Filtrar por talla en el inventario

En la lista de productos, un nuevo filtro "Talla" permitirá escribir o elegir "M" y ver todos los productos que tienen stock en talla M. La query navega: atributo_valor (M) → producto_variante_valor → producto_variante → stock_variante_sede. Resultado: todos los productos donde hay talla M con stock > 0.

10. Preguntas pendientes

Antes de empezar a construir, necesito confirmar estas decisiones contigo:

❓
1. ¿Los productos pueden estar en solo 1 categoría o en múltiples?
Mi recomendación: 1 sola (hoja del árbol). Para clasificación adicional, usar las etiquetas que ya existen.
❓
2. ¿Se maneja stock por variante (talla+color) POR SEDE, o un stock total del negocio?
El sistema actual ya tiene stock por sede. Lo natural es extenderlo: stock de "Polo M Rojo" en Lima = 5, en Callao = 3. ¿Confirmas este nivel de granularidad?
❓
3. ¿Las etiquetas actuales (planas, por negocio) conviven o las reemplazamos con categorías?
Mi recomendación: conviven. Categorías = clasificación principal (árbol global). Etiquetas = tags libres del negocio (ej: "Temporada verano 2026", "Liquidación").
❓
4. ¿El catálogo público de Next.js debe mostrar el selector de variantes (talla/color) con cambio de imagen?
Asumo que sí, ya que lo mencionaste. ¿Es parte de este sprint o se planifica después?
❓
5. ¿Los movimientos de inventario (INGRESO/SALIDA) siempre requerirán especificar variante, o solo cuando el producto tenga variantes configuradas?
Mi recomendación: condicional. Si el producto tiene variantes, el movimiento DEBE incluir variante_id. Si no, usa el flujo actual.
❓
6. ¿Necesitas que el sistema genere automáticamente TODAS las combinaciones de variantes, o el negocio las crea de a una?
Ej: si elige tallas S, M, L y colores Blanco, Negro → ¿el sistema crea las 6 combinaciones automáticamente? Lo recomiendo, con opción de eliminar las que no existen (ej: "no tenemos XL en color Rojo").
❓
7. ¿El negocio puede agregar colores o tallas que no están en el seed global?
Ej: un negocio tiene una talla "Talla Única" o un color "Verde Esmeralda Oscuro". ¿Permitimos valores custom por negocio o solo los del sistema?
